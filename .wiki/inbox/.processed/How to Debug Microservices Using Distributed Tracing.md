
# [How to Debug Microservices Using Distributed Tracing](https://medium.com/@niteshthakur498/how-to-debug-microservices-using-distributed-tracing-f4df418e9e66)

Debugging a monolith is frustrating. Debugging microservices without the right tools is a different kind of painful.

In a monolith, when something goes wrong, you check the logs. One log file, one process, one timeline. You find the exception, trace it back to the line that caused it, and fix it. The whole thing might take twenty minutes.

In a microservices system, a single user request might travel through an API gateway, an auth service, an order service, an inventory service, a payment service, and a notification service — each running as a separate process, each writing to its own logs, potentially on different machines. When that request fails, you’re left asking: which service failed? Where in the chain? Was it slow, or did it error? Was the slow part my service or the one upstream?

Log files from six different services, none of which share a common identifier for that specific request, don’t answer those questions easily. You end up correlating timestamps, guessing at causality, and spending an hour on something that should take five minutes.

Distributed tracing exists specifically to fix this. It gives you a way to follow a single request across every service it touches, see exactly how long each step took, and pinpoint failures — without manually correlating logs across multiple systems.

## The Core Concept: Traces, Spans, and Context Propagation

Before jumping into tools and setup, it’s worth understanding what distributed tracing is actually tracking.

A **trace** represents a single end-to-end request as it flows through your system. Think of it as the complete story of one user action — from the moment it hits your API gateway to the moment a response is returned.

A **span** is one unit of work within that trace. Each service that handles the request contributes one or more spans. A span has a name, a start time, a duration, and a status (success or error). Spans can be nested — if the order service calls the inventory service, the inventory service’s span is a child of the order service’s span.

Put together, a trace is a tree of spans. Visualized, it looks like a Gantt chart:

```c
[API Gateway          ────────────────────────────────── 320ms]
  [Auth Service       ──── 45ms]
  [Order Service      ──────────────────── 240ms         ]
    [DB Query         ──── 30ms]
    [Inventory Service ─────────── 180ms                 ]  ← slow
      [DB Query       ──── 25ms]
    [Payment Service  ── 20ms]
  [Notification Service ── 15ms]
```

At a glance, you can see the Inventory Service is where most of the time is going. Without this view, you’d be tailing logs and guessing.

The mechanism that makes this work across service boundaries is **context propagation**. When Service A calls Service B, it passes a small piece of metadata in the HTTP headers — a trace ID (identifying the overall request) and a parent span ID (identifying which span in Service A made the call). Service B reads these headers, creates a child span, and continues the trace. This chain of IDs is what links all the spans together into a single trace, even though they were generated by completely different processes.

The W3C Trace Context standard (`traceparent` header) is the modern, interoperable format for this propagation. Most tracing libraries support it out of the box.

## Setting Up Distributed Tracing in Spring Boot

![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*tR-VswD5tm0hw7JBglpnDg.png)

The modern approach to distributed tracing in Java uses **OpenTelemetry** — an open standard and SDK for generating, collecting, and exporting telemetry data. It’s vendor-neutral, which means you can switch your tracing backend without changing application code.

For Spring Boot, the setup is surprisingly straightforward.

**Add the dependencies:**

```c
<!-- Spring Boot Actuator (required) -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>

<!-- Micrometer Tracing with OpenTelemetry bridge -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-otel</artifactId>
</dependency>

<!-- OpenTelemetry exporter — sends spans to your backend -->
<dependency>
    <groupId>io.opentelemetry</groupId>
    <artifactId>opentelemetry-exporter-otlp</artifactId>
</dependency>
```

**Configure in** `**application.yml**`**:**

```c
management:
  tracing:
    sampling:
      probability: 1.0   # 1.0 = trace every request (use lower in high-traffic prod)
  otlp:
    tracing:
      endpoint: http://localhost:4318/v1/traces  # your collector endpoint

spring:
  application:
    name: order-service   # shows up as the service name in traces
```

That’s the baseline. Spring Boot’s auto-configuration handles the rest — it instruments incoming HTTP requests, outgoing `RestTemplate` and `WebClient` calls, and Spring Data repository operations automatically. You don't need to manually create spans for these common operations.

**Run a tracing backend locally with Docker:**

```c
# Jaeger — an open source, full-featured tracing backend
docker run -d --name jaeger \
  -p 16686:16686 \   # Jaeger UI
  -p 4318:4318 \     # OTLP HTTP receiver
  jaegertracing/all-in-one:latest
```

Open `http://localhost:16686` and you'll see the Jaeger UI. Make a request to your Spring Boot service, search by service name, and your first trace appears. Each span is visible, with timing, tags, and any errors.

## Adding Custom Spans

The auto-instrumentation covers most ground, but sometimes you want to trace a specific piece of business logic — a complex calculation, an external API call that doesn’t use `RestTemplate`, or a batch operation inside a service method.

```c
import io.micrometer.tracing.Tracer;
import io.micrometer.tracing.Span;

@Service
public class OrderService {
    private final Tracer tracer;
    private final InventoryClient inventoryClient;
    public OrderService(Tracer tracer, InventoryClient inventoryClient) {
        this.tracer = tracer;
        this.inventoryClient = inventoryClient;
    }
    public OrderResult placeOrder(OrderRequest request) {
        // Custom span for the inventory check
        Span inventorySpan = tracer.nextSpan()
                .name("inventory.check")
                .tag("order.id", request.getOrderId())
                .tag("item.count", String.valueOf(request.getItems().size()));
        try (Tracer.SpanInScope scope = tracer.withSpan(inventorySpan.start())) {
            InventoryResult result = inventoryClient.checkAvailability(request.getItems());
            inventorySpan.tag("inventory.available", String.valueOf(result.isAvailable()));
            return processOrder(request, result);
        } catch (Exception e) {
            inventorySpan.error(e);   // marks the span as failed with the exception
            throw e;
        } finally {
            inventorySpan.end();
        }
    }
}
```

Tags are key-value pairs attached to a span. They’re how you add business context to your traces — not just “this span took 200ms” but “this span was checking inventory for order ORD-12345 with 3 items.” When you’re debugging a slow request, those tags let you filter and search traces by business dimensions, not just by service name.

The `inventorySpan.error(e)` call is important — it marks the span as an error and attaches the exception to it. In the trace UI, errored spans are highlighted in red. You can see at a glance which span in a trace failed without reading any log output.

## Reading a Trace: What to Look For

Once your traces are flowing into the backend, knowing what to look for is half the skill.

**Find the slow span first.** The Gantt chart view makes this immediate — look for the longest bar. If the Inventory Service spans takes 180ms out of a 200ms total request, that’s where to start. Everything else is noise until you understand that.

**Look at the critical path.** The critical path is the sequence of spans that determines the total request duration. Spans that run in parallel don’t add to the total — only the sequential chain matters. If two service calls run simultaneously, their combined contribution to total time is the slower of the two, not the sum.

**Check for unexpected serialization.** Sometimes what looks like one long span is actually many short sequential calls to a database or external service — the classic N+1 query pattern made visible. One span for “load order” followed by 20 spans of “load order item” tells you something you’d never see in aggregated metrics.

**Follow error propagation.** When a request fails, the error usually originates in one span and propagates back up through parent spans. Tracing shows you the origin — the span that first errored — rather than just the outermost service that reported the failure to the user.

**Use tags to filter.** If you’re investigating reports of slow requests for a specific customer, search traces by `customer.id` tag (if you've added it). You don't need to dig through all traces — just the ones relevant to the customer's experience.

## Connecting Traces to Logs

Distributed tracing is most powerful when your traces and your logs are connected. When you’re looking at a failing span, you want to jump directly to the relevant log lines — not search through log output manually.

The standard approach is to include the trace ID and span ID in every log line. With Spring Boot and Micrometer Tracing, this happens automatically when you use SLF4J:

```c
2024-01-15T14:32:07Z  INFO [order-service,4bf92f3577b34da6a3ce929d0e0e4736,a3ce929d0e0e4736] OrderService - Processing order ORD-12345
```

The values in brackets are: service name, trace ID, span ID. If you’re shipping logs to a log aggregator (Elasticsearch, Loki, Datadog), you can search by trace ID and see every log line that was emitted during that specific request — across all services.

```c
# In application.yml — include trace context in log output
logging:
  pattern:
    console: "%d{yyyy-MM-dd'T'HH:mm:ssZ} %5p [${spring.application.name},%X{traceId},%X{spanId}] %logger{36} - %msg%n"
```

This gives you bidirectional navigation: from a trace, jump to its logs; from a log line, find its trace. The two views are complementary — traces show you the shape of the request, logs show you the detail of what happened inside each span.

## Sampling: Not Every Request Needs to Be Traced

![](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*OqSM4MEWxgz18MpFOBePmA.png)

In development, sampling at `1.0` (every request gets traced) is fine. In production with thousands of requests per minute, tracing everything is expensive in terms of storage and processing cost.

Sampling is how you control what percentage of traffic gets traced. There are a few strategies:

**Probability-based (head sampling):** trace a fixed percentage of requests. Setting `probability: 0.1` means 10% of requests generate a trace. Simple, low overhead, but you might miss the one slow request in the 90% that wasn't sampled.

**Always sample errors:** regardless of the overall sampling rate, trace every request that results in an error. Most tracing backends and libraries support this. It ensures you always have trace data for failures — the requests where you most need it.

**Tail-based sampling:** make the sampling decision *after* the request completes, based on what happened. Slow requests get sampled, errored requests get sampled, normal fast requests don’t. This is the most accurate approach but requires a collector that can buffer and evaluate spans before deciding what to keep. OpenTelemetry Collector supports this with the `tail_sampling` processor.

For most teams getting started, probability-based sampling at 10% plus always-sample-on-error covers the practical need at reasonable cost.

## Common Mistakes Worth Avoiding

**Not propagating context through async operations.** If you kick off a background thread or use `CompletableFuture` without explicitly propagating the trace context, the child spans won't be linked to the parent trace. Micrometer Tracing provides context-aware wrappers for common async patterns — use them.

```c
// Wrong — loses trace context
CompletableFuture.runAsync(() -> doWork());

// Right — propagates context into the async call
CompletableFuture.runAsync(tracer.currentTraceContext().wrap(() -> doWork()));
```

**Missing service names.** Every service must set `spring.application.name`. Without it, all your spans appear under a generic name in the trace UI and you lose the ability to filter by service.

**Tracing but not tagging.** Traces without tags are a Gantt chart without labels. Add business-relevant tags — order IDs, customer IDs, item counts — to the spans that matter. Tags are what make a trace searchable and useful when you’re in debugging mode.

**Ignoring trace IDs in customer support workflows.** If your system logs trace IDs and your user-facing errors include a reference ID, support teams can look up the exact trace for a specific complaint. This transforms “the customer says something was slow on Tuesday” into a five-second lookup.

## The Before and After

Without distributed tracing: a user reports that their order took 40 seconds to confirm. You look at the order service logs, see nothing unusual. You check the payment service — fine. You check the inventory service — also fine. You spend two hours correlating timestamps across log files and eventually suspect a network issue that you can’t confirm because the data isn’t there.

With distributed tracing: you search for the user’s request by their customer ID tag. You find the trace. The Gantt chart shows the inventory service taking 38 seconds, with a single database query span inside it taking 37 of those seconds. You find the query, check the execution plan, discover a missing index. You add the index. The problem is gone in forty minutes total.

The difference isn’t just speed. It’s certainty. You’re not guessing at the cause — you’re reading it directly from the trace.

Distributed tracing doesn’t make microservices debugging *easy*. It makes it *tractable*. And in a system with ten services and thousands of requests per minute, tractable is exactly what you need.
